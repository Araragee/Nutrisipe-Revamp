## 4. Improvements & redesign strategy

### A. Architectural cleanup

- **One API client.** Pick `src/http/endpoints/*`, delete `src/api/*` and the `useRecipes` composable. Standardize response shape on `{ success, data, pagination }` end-to-end (the backend already does this; the FE has three competing shapes).
- **One `httpClient`.** Merge `src/http/client.ts` and `src/api/client.ts` into a single instance with shared interceptors.
- **One layout.** AppShell is canonical; delete the leftover sidebars and nav.
- **Type the Recipe payload.** Lock `RecipeIngredient = { name, quantity, unit? }` (rename `qty`→`quantity`), `RecipeInstruction = { step, text }`, `RecipeNutrition = { calories, protein, carbs, fat, fiber? }`. Migrate the wizard, the renderers, and the seeder together so JSON in DB matches.
- **ID type discipline.** Backend uses UUID strings (Prisma), but `src/api/recipes.ts` still types IDs as `number`. Sweep all `id: number` → `string`.

### B. Design-system consolidation

- The revamp uses OKLCH tokens, glass borders, and Montserrat. Anything still using `bg-gray-100`, `dark:bg-gray-800`, raw hex, or Tailwind `gray-*`/`orange-500` numeric scales is pre-revamp leakage — `SettingsView`, `LayoutSidebar`, `LayoutRightSidebar`, `NotificationBell`, parts of `MessagesView`. Migrate to tokens (`bg-background-secondary`, `border-glass-border`, `text-text-dim`, etc.) so the look is consistent.
- Extract the duplicated card / hero / tab patterns into `BaseCard`, `BaseHero`, `BaseTabs` so future views (e.g., for Variations and Collections) don't reinvent them.

### C. Feature coverage gaps to design for

1. **Comments + Ratings on the recipe surface** — backend is fully ready (`/api/comments`, `/api/ratings`); the modal and detail view just don't mount them.
2. **Real-time notification bell** — `socket.io` is connected on login, but the bell never subscribes. Hook `notification:new` to bump `unreadCount`.
3. **Following-only feed** — the recommendation logic exists in `recommendationService`; expose it.
4. **Recipe Variations** — components exist (`ForkRecipeButton`, `VariationsList`, `VariationChain`) but nothing surfaces them. Add a "Variations" tab on `RecipeDetailView` and a Fork CTA next to Save/Share.
5. **Collections / Saved sub-folders** — backend has full Collection model + endpoints, frontend SavedRecipesView ignores it. Build a "My Collections" view at `/saved` with grouped boards.
6. **Mentions** — `@user` autocomplete component exists (`MentionInput`) but not used in the create-post modal.
7. **Onboarding → preferences-aware feed** — once preferences land, weight `recommendationService` toward selected diets/goals.

### D. Performance / quality

- `searchPostsHandler` filters tags via `contains` of a JSON string — works for SQLite dev but won't scale; use a real `tags` join table or Postgres array when you migrate off SQLite.
- `feed` fetches all likes/saves in two follow-up queries; that's fine, but add eager `_count` loading instead of separate counts in admin dashboard.
- `RecipeModal` and `PinCard` reach for `picsum.photos` and hardcoded `http://localhost:3001/${imageUrl}` — externalize the API origin into `import.meta.env.VITE_API_URL.replace('/api','')`.
- Add Vue `<Suspense>` skeletons (the TODO in HomeView).
- Add `vue-tsc` to CI; right now broken types like `id: number` vs UUID strings slip in.
